# 上海买房地图 — 设计文档

日期：2026-07-17
类型：个人买房研究工具，分两阶段：先本地完整版，后上云多人共享

## 实施阶段（重要）

本项目分两阶段推进，先交付可用的本地版，再扩展为云端共享版：

- **第一阶段 · 本地完整版（先做，无需任何账号/云）**：在 Mac 上做出完整可用的地图工具——全量抓取全市板块、全量查看、勾选板块、收藏、新房/二手区分、C 风格界面、本地轻服务 + 双击启动、iCloud 跨设备同步（限苹果设备）。目的：最快拿到能用的东西，并**尽早验证最不确定的环节**（官方网签数据能否抓取、板块边界 GeoJSON 能否获得）。
- **第二阶段 · 上云多人共享（后做，需 GitHub + 部署平台）**：把工具搬上云，让闺蜜（安卓）、父母（Windows）等在自己设备、自己网络下打开网址即可使用并**实时收藏**。引入后端 + 数据库 + 部署。数据分层：本地全量探索，**勾选“发布”**精选板块到云端展示；收藏为云端实时读写的轻数据。多人用**无密码身份选择**（如小美/阿珍/爸妈）。

> 需求演进说明：项目最初设想为“本地个人小工具”，但因需覆盖多人、多设备（安卓/Windows）、外出手机使用、实时收藏，最终必然走向“部署上线的小网站”。故拆两阶段：第一阶段满足自用并验证数据可行性；第二阶段满足共享。第二阶段的平台选择（GitHub / Vercel+Supabase 等）待用户准备后确认。

## 目标

一个交互式的上海市区地图，帮助研究买房。地图上能看到：

- 全部行政区的位置与划分（大区结构）
- 全市齐全的房产板块（可悬停高亮、点击查看详情）
- 明显区分内环 / 中环 / 外环三条环线
- 前期全市全量铺开查看，据此筛选出重点关注板块

点击某个板块，展示该板块的总结（新房/二手房均价分开、优点、缺点）以及可出售小区列表（小区名、类型、均价、最近成交、房龄、开发商、对口学校等）。

## 整体架构

### 第一阶段架构（本地完整版）

项目由三部分组成：

1. **地图应用（前端）** — 日常查看的界面。`index.html` + JS + CSS，用 Leaflet 渲染。读取本地地理边界（GeoJSON）、房产数据（`housing.json`）和收藏档案（`profiles.json`）。视觉基调：方案 C（精致轻奢，中性底 + 小面积彩色点缀）。
2. **本地轻服务（`server.py`）** — 用 Python 标准库实现的小服务：把前端页面/数据发给浏览器；接收前端的收藏/改名/切换/**勾选发布**请求，写入本地文件。仅本机运行。通过双击 `start.command` 启动。
3. **抓取脚本（`scrape.py`）** — 手动运行，**全量抓取**（全市板块，供探索），产出 `housing.json`。与前端解耦——脚本失败时地图仍用旧数据正常显示。

关键原则：**地图与数据解耦**。地理边界做好基本不动；变的是 `housing.json`（抓取产出）与 `profiles.json`（收藏/档案/发布标记写入）。抓取只拿到部分数据时，地图仍能正常展示已有部分。收藏跨设备同步（限苹果设备）依赖将 `data/` 放入 iCloud（见“iCloud 跨设备同步”）。

### 第二阶段架构（上云共享版，概要）

在第一阶段之上扩展，不推翻已有成果：

- **前端**：沿用同一套页面（响应式，适配手机/平板/桌面；可做成 PWA“添加到主屏幕”）。
- **后端 + 数据库**：部署到云（如 Vercel + Supabase/Postgres）。存两类数据：① **精选房产数据**（只读展示，来自本地“发布”）；② **收藏与多人档案**（实时读写的轻数据）。
- **数据分层**：本地 Mac 全量抓取 + 全量查看 + **界面勾选要公开的板块 → 点“发布” → 仅将选中板块数据推送到云数据库**（方式 A：可视化勾选发布）。云端始终只放精选，数据量小、聚焦、在免费额度内。
- **多人**：无密码身份选择（打开网址选“我是谁”，各看各的收藏），沿用 `profiles` 概念搬到云端。
- **家人**：闺蜜（安卓）、父母（Windows）打开网址即可看精选 + 实时收藏，无需装任何东西。
- **抓取**：仍在本地 Mac 跑（全量），云端不做抓取。

> 云化收藏回流：外出手机看的是发布后的在线精选版，收藏实时写云数据库，天然全员同步，无需 iCloud 中转（iCloud 同步仅用于第一阶段本地苹果设备间）。

## 数据结构

### 地理文件（一次性制作，基本不动）

- `districts.geojson`：行政区多边形。属性含区名。
- `blocks.geojson`：板块多边形（用街道边界近似真实房产板块）。属性含：板块名、所属行政区、所属环线（内环 / 中环 / 外环）。
- `rings.geojson`：内/中/外环三条环线，纯参考图层。

### 房产数据（抓取脚本产出，可反复更新）

`data/housing.json`：

```json
{
  "updated_at": "2026-07-17",
  "blocks": {
    "联洋": {
      "resale_avg_price": 78000,
      "new_avg_price": 95000,
      "pros": ["学区好", "商业成熟"],
      "cons": ["房龄偏老", "单价高"],
      "has_favorite": true,
      "communities": [
        {
          "name": "仁恒森兰雅苑",
          "property_type": "resale",
          "avg_price": 82000,
          "last_deal": "2026-06",
          "built_year": 2008,
          "favorite": true,
          "developer": "仁恒置地",
          "school": {
            "name": "XX实验小学",
            "source": "浦东新区教育局",
            "year": 2026,
            "note": "仅供参考，以当年招生政策为准"
          }
        },
        {
          "name": "森兰名轩",
          "property_type": "new",
          "list_price": 96000,
          "built_year": 2027,
          "developer": "XX地产"
        }
      ]
    }
  }
}
```

- 顶层 `updated_at`：数据更新日期，界面角落展示。
- 板块的 `pros`/`cons` 是**多个评价源汇总去重后的一份综合优缺点**（不分来源），新增评价源自动并入。
- 板块的 `resale_avg_price`（二手房成交均价）与 `new_avg_price`（新房均价）**分开统计、分别显示**；某类无小区则该字段缺省。**地图板块着色默认按 `resale_avg_price`**（更反映真实市场），新房均价在面板作补充。
- `blocks` 以**板块名为 key**，与 `blocks.geojson` 的板块名对应关联。
- 单价单位：元/㎡。价格主源为官方“网上房地产”/住建局网签数据。
- 小区 `property_type`：`"new"`（新房）或 `"resale"`（二手房），用于筛选与详情标签。
- 小区价格字段按类型不同：二手房用 `avg_price`（成交均价）+ `last_deal`（最近成交）；新房用 `list_price`（开盘/备案价），通常无 `last_deal`。新房字段缺失是常态，前端“有则显示、无则跳过”。
- `built_year`：建成年份（整数，如 `2008`）；前端据此自动换算显示“房龄 X 年”。新房为未来年份/在建时，前端特殊显示（如“新房 · 预计 2027 交付”）。
- 小区字段**均为可选**：前端有则显示、无则跳过。`developer`、`school` 属第二批数据源，第一批可缺省。
- `school` 是“对口学校（仅供参考）”定位，非“是/否学区房”的打包票标签，须带 `source`（数据来源）与 `year`（年份）标注。
- 小区的 `favorite: true` 标记该小区是关注小区（来自抓取配置 `FOCUS_COMMUNITIES` 或收藏）；界面据此置顶 + 星标高亮。
- 板块的 `has_favorite: true` 表示该板块下含关注小区；界面据此在地图上给板块加标记，即便板块本身无其他数据。
- 没有房产数据的板块（如外环外）照常显示轮廓，点开显示"暂无数据"。

### 收藏档案数据（前端点星写入，抓取脚本读取）

`data/profiles.json` —— 支持多人各自独立收藏，文件名固定，人名为字段（不随改名重命名文件，保证 iCloud 同步稳定）：

```json
{
  "profiles": [
    { "id": "user1", "name": "小美", "blocks": ["联洋"], "communities": ["仁恒森兰雅苑"] },
    { "id": "user2", "name": "阿珍", "blocks": ["碧云"], "communities": [] }
  ],
  "active": "user1"
}
```

- 每个档案有稳定的 `id`（内部标识，不变）、`name`（显示名，可在网页自由修改，仅改此字符串）、以及各自的 `blocks`/`communities` 收藏。
- `active`：当前选中的档案 id。前端顶部提供身份切换（显示各档案 `name`），切换即切换收藏与“只看收藏”视图。
- 点星收藏/取消、改名、切换身份，均通过 `server.py` 写入此文件；文件名恒为 `profiles.json`，改名只改 `name` 字段，收藏数据原地不动。
- 抓取脚本读取此文件，将**所有档案**的收藏**合并去重**后纳入抓取范围（房产数据共享，仅展示时按当前档案过滤）。
- `housing.json` 中的 `favorite`/`has_favorite` 按合并后的收藏在抓取时标注。
- 无密码保护（两人一起看房、不互相保密，见架构决策）。

## 交互与界面

三层叠加（从下到上）：

1. **环线**：内/中/外环虚线圈，参考层。
2. **行政区**：粗边界 + 淡底色 + 大区名，一眼可辨大区范围（浦东/闵行/徐汇…）。
3. **板块**：可交互色块，按环线分三档色阶配色；中环内重点板块颜色更实。板块着色的均价默认取**二手房成交均价**（`resale_avg_price`）。

交互：

- **新房/二手房筛选**：界面提供筛选器「全部 / 只看新房 / 只看二手房」，**两处联动**——地图上只高亮含对应类型小区的板块；点开板块后小区列表只列对应类型。板块面板均价按筛选相应突出（二手成交均价 / 新房均价）。
- **悬停板块**：整块高亮 + 轻微放大 + 小提示（板块名 + 均价）。
- **点击板块**：右侧滑出面板，显示板块总结（二手房成交均价 + 新房均价分开显示、优点、缺点）+ 可出售小区列表。每个小区显示：小区名、**类型标签（新房/二手房）**、价格（二手房显示成交均价+最近成交，新房显示开盘/备案价）、**房龄（据 `built_year` 换算，新房显示交付年份）**、开发商、对口学校（仅供参考，附来源/年份）——字段有则显示、无则跳过。
- **关注小区标记**：`favorite` 小区在面板列表中置顶 + 星标高亮；含关注小区的板块（`has_favorite`）在地图上带标记，即便板块本身无其他数据。
- **身份切换（多人档案）**：界面顶部下拉显示各档案名（如“小美 / 阿珍”），切换即切换当前收藏与“只看收藏”视图。档案名可在此就地改名（仅改 `profiles.json` 的 `name` 字段）。
- **收藏（点星）**：板块面板和小区列表旁有星标☆，点击即收藏（★）到**当前档案**，再点取消。收藏动作经 `server.py` 写入 `profiles.json`，经 iCloud 跨设备同步。
- **只看收藏**：界面有一个开关，打开后地图/列表只重点显示当前档案已收藏的板块和小区。
- **角落信息**：图例 + “数据更新于 YYYY-MM-DD”。

视觉实现将使用 frontend-design 方法，避免套模板的通用观感。

## 抓取脚本

### 抓取范围配置（可调整）

范围配置写在脚本顶部，支持两种模式，可单用或叠加（取并集）：

```python
# ===== 抓取范围配置 =====

# 模式一：按环线圈（前期不了解具体板块时用）
FOCUS_RINGS = ["内环", "中环"]   # 抓这些环线内的所有板块；设为 [] 则不启用

# 模式二：按具体板块（后期锁定重点时用）
FOCUS_BLOCKS = ["联洋", "古北"]  # 精确指定；设为 [] 则不启用

# 模式三：按具体小区（对某个楼盘单独感兴趣时用）
FOCUS_COMMUNITIES = ["仁恒森兰雅苑"]  # 精确抓这些小区，哪怕所在板块没被选中；设为 [] 则不启用
```

用法：

- **前期广撒网**：`FOCUS_RINGS = ["内环", "中环"]`、其余 `[]` → 抓中环以内所有板块。
- **后期精抓**：`FOCUS_RINGS = []`、`FOCUS_BLOCKS = [...]` → 只抓锁定的几个板块，更快更省。
- **盯具体楼盘**：在 `FOCUS_COMMUNITIES` 列出小区名 → 单独抓这些小区，归入其所属板块，并在界面标为关注。
- **混合**：三者可任意叠加，取并集。

**收藏内容默认纳入**：抓取脚本还会读取 `data/profiles.json`，把**所有档案**收藏的板块/小区**默认加入本次抓取范围**。最终抓取范围 = 各档案收藏并集 ∪ `FOCUS_RINGS` 命中板块 ∪ `FOCUS_BLOCKS` ∪ `FOCUS_COMMUNITIES`。这样界面上点星收藏的对象，无需再改脚本配置就会被自动刷新。

按环线筛选依赖 `blocks.geojson` 中每个板块的“所属环线”属性：脚本先按环线属性算出板块清单，合并 `FOCUS_BLOCKS`，叠加 `FOCUS_COMMUNITIES` 与 `profiles.json` 各档案的收藏项，去重后抓取。收藏及 `FOCUS_COMMUNITIES` 命中的小区在产出中标 `favorite: true`，其所属板块标 `has_favorite: true`。改配置或改收藏后重新运行脚本即可，不碰其他代码。

### 多源可插拔架构

抓取不写成一整块，而是拆成一组**独立的数据源模块**，每个模块产出 `housing.json` 中的一部分字段片段。主流程 `scrape.py` 决定抓哪些板块/小区，依次调用已注册的模块，把各模块返回的片段**合并**成完整记录。某个源因反爬变化而失效时，只影响它贡献的字段，不会拖垮整体。

字段与模块是**多对多**关系，合并方式分两类：

- **独占字段（单值，多源 fallback 链）**：如 `avg_price`、`developer`、`school`——只需一个正确答案。可为其配置**按优先级排序的多个源**，主流程**按序尝试、取第一个成功的**（方案 A）：主源抓到就用主源，主源失败/无数据才依次降级到备用源，全部失败则留空（前端不显示）。示例——成交价：官方网上房地产（主）→ 贝壳（备1）→ 安居客（备2）。保证优先采用最权威的源，又有兜底。不全抓所有源，省请求、对目标站更温和。
- **汇总字段（多值，多源合并去重）**：如板块的 `pros`/`cons`（优缺点）。多个评价源（贝壳、将来的安居客/小红书等）**全部抓取**，各自产出观点列表，主流程**汇总去重成板块统一的一份**综合评价（不分来源）。越多源越全。

两种模式的区别：独占字段是“**择优兜底**”（按序取一），汇总字段是“**兼收并蓄**”（全取合并）。

分批实现：

- **第一批（核心链路）**：`price_official`（成交价 ← 官方网上房地产/住建局，主源）+ `review_beike`（评价 ← 贝壳点评）。目标是把“抓取 → housing.json → 地图展示”全链路跑通。
- **第二批（增量）**：成交价备用源（贝壳/安居客 fallback）+ `developer`（开发商）+ `school`（对口学校，仅供参考）+ 更多**评价源**（汇入统一 `pros`/`cons`）。往骨架里插入，不改动已有模块。

各数据源可靠性/难度参考（成交价以官方网签为准锚，其余为交叉验证/补充）：官方网上房地产 ★★★★★、贝壳/链家 ★★★★（字体加密+反爬强）、安居客/中原/我爱我家 ★★★、房天下 ★★（需甄别）。评价先用贝壳点评，安居客/小红书/篱笆网/宽带山等作为后续可插拔的补充评价源。

### 抓取行为

- Python 脚本，仅抓取范围配置（含收藏）命中的板块/小区。
- 每个小区带 `property_type`（新房/二手房）与 `built_year`，由价格/详情源在抓取时判定填入（新房源与二手房源可能不同，均归入同一小区记录）。
- **板块级均价为派生值**：主流程按小区 `property_type` 分组，分别聚合出板块的 `resale_avg_price`（二手房）与 `new_avg_price`（新房），不把两类价格混算。
- 温和策略：请求间加随机延时、使用正常 UA、低频运行。
- **原子更新保护**：先写 `housing.json.tmp`，本轮抓取完整跑完后再重命名覆盖 `housing.json`；进程中途异常退出则丢弃 tmp，上次的好数据不受影响。单个数据源抓不到属正常（独占字段走 fallback、汇总字段少一个源），不视为失败，对应字段留空即可。
- 官方网签数据以外若涉及字体加密价格（贝壳等），在实现该模块时具体攻克；先保证流程与数据结构跑通，必要字段必要时可人工校对。

### 如何新增数据源/字段（插拔指南）

面向后续开发者。有三种新增场景，都无需改动已有模块：**(甲) 新增独占字段**（如开发商、学校）；**(乙) 为独占字段新增备用源**（fallback，如给成交价加贝壳兜底）；**(丙) 为汇总字段新增数据源**（如给评价再加安居客/小红书）。

**1. 模块约定** — 在 `scraper/sources/` 新建模块，实现统一签名的 `fetch()`：

```python
# (甲) 独占字段示例：scraper/sources/developer.py
def fetch(community: dict) -> dict:
    """返回本模块负责的字段片段；抓不到返回 {}，绝不抛异常。"""
    return {"developer": "仁恒置地"}

# (乙) 独占字段的备用源示例：scraper/sources/price_beike.py
def fetch(community: dict) -> dict:
    """产出与主源相同的字段（avg_price/last_deal）；仅在主源缺该字段时被采用。"""
    return {"avg_price": 81000, "last_deal": "2026-05"}

# (丙) 汇总字段（评价）示例：scraper/sources/review_anjuke.py
def fetch(community: dict) -> dict:
    """评价源统一产出 pros/cons 观点列表，主流程负责跨源汇总去重。"""
    return {"pros": ["近地铁", "物业好"], "cons": ["车位紧张"]}
```

约定：
- 独占字段模块只填自己的字段，不覆盖别人。
- 失败时返回空 dict 并记录日志，不抛异常（单源失效不拖垮整体）。
- 自带温和抓取策略（延时、UA）。

**2. 注册模块** — 独占字段配置成**按优先级排序的 fallback 链**（主流程按序取第一个非空值）；汇总字段列成一组（主流程全抓合并去重）：

```python
# 独占字段：优先级列表，主源在前，备用源在后（fallback）
FIELD_SOURCES = {
    "price":     [price_official, price_beike, price_anjuke],  # 官方→贝壳→安居客
    "developer": [developer],
    "school":    [school],
}
# 汇总字段：全部抓取、合并去重
REVIEW_SOURCES = [review_beike, review_anjuke]                 # 评价多源
```

- 给独占字段加备用源（乙）：在对应字段的列表**追加一行**（放在主源之后）。
- 给评价加源（丙）：在 `REVIEW_SOURCES` **追加一行**。
- 主流程逻辑：独占字段按列表顺序尝试、取第一个成功的；评价源全抓、`pros`/`cons` 汇总去重。新增源均不改主流程。

**3. 约定数据结构** — 若引入新字段，在本文档“数据结构 › 房产数据”补充说明，保持**字段可选**（前端有则显示、无则跳过）。带参考性质的字段（如学校）须附 `source` 与 `year`。为已有字段加备用源不改结构（产出同名字段）；汇总字段（`pros`/`cons`）也不改结构。

**4. 前端渲染** — 前端按“字段存在才渲染”展示；新增独占字段通常无需改前端结构，如需专门样式（如学校“仅供参考”提示）加一处条件渲染。新增备用源、新增评价源对前端**完全透明**（仍是同名字段 / 同一份 `pros`/`cons`）。

**5. 跑一次验证** — 双击 `update-data.command`，确认 `housing.json` 中该字段命中率提升 / 评价更全、地图面板正确显示。

完整示例见 `sources/developer.py`、`sources/school.py`（独占字段）、`sources/price_*.py`（独占字段 fallback 链）与 `sources/review_*.py`（汇总字段）。

### 预期管理（诚实说明）

抓取能否稳定拿到全部字段，取决于目标站点当时的反爬强度，无法 100% 保证。但因地图与数据解耦、数据源可插拔，即使部分源失效，地图仍能正常展示已有字段。学校字段为“对口学校（仅供参考）”，受当年招生政策/多校划片影响，不构成买房决策的唯一依据。抓取属于个人小范围、低频使用。

## 数据更新流程

看地图和更新数据是**两条相互独立的流程**，没有强制先后：看地图不要求先抓数据（读已有 `housing.json`）；更新数据不要求先开服务（`scrape.py` 独立运行）。唯一关联是：更新后需刷新浏览器才显示新数据。

### 看地图（日常）

双击 `start.command` → 自动起服务并打开浏览器到 `http://localhost:8000`。读取已有数据，无需先抓取。

### 更新数据（偶尔，如每周一次）

双击 `update-data.command` → 运行抓取脚本（默认含收藏内容 + 手动 `FOCUS_*` 配置），**跑完停住显示摘要**（本次更新板块数、失败项、更新时间），确认后手动关闭窗口。因反爬可能导致个别板块抓取失败，摘要便于察觉。抓取完成后，刷新已打开的地图页面即可显示新数据。

也可命令行运行：`cd scraper && python3 scrape.py`。

### 调整关注范围

关注范围有两个来源，二者取并集：

1. **界面收藏（推荐日常用）**：在地图上点星收藏/取消板块或小区（写入当前档案），存 `profiles.json`，下次抓取自动纳入。无需改代码。
2. **脚本配置（批量/环线用）**：编辑 `scraper/scrape.py` 顶部的 `FOCUS_RINGS` / `FOCUS_BLOCKS` / `FOCUS_COMMUNITIES`，适合"抓整个内环"这类批量范围。

改动后重新运行抓取（双击 `update-data.command`）、刷新浏览器即可。若想关注的板块地图上根本没有轮廓，需在 `blocks.geojson` 补边界多边形（设计上板块全市齐全，正常无需此操作；确有缺失请告知板块名）。

## iCloud 跨设备同步

需求：单人多设备（家/公司/手机）看到同一份收藏；两人（如“小美/阿珍”）各自独立收藏。不需要注册/登录/密码。

方案：**多人档案（本地身份切换）+ iCloud 文件同步**，避开账号体系与云端后端。

- **多人** = `profiles.json` 里多个档案，界面切换身份（见“收藏档案数据”）。无密码——两人一起看房、不互相保密。
- **跨设备** = 把项目的 `data/` 目录（或整个项目）放在 iCloud Drive 同步文件夹内。各设备的地图应用读写同一份 `profiles.json`，iCloud 自动同步。因文件名固定、改名只改字段，同步不会因重命名产生冲突。

### 配置步骤（实现完成后带用户操作）

1. 将项目移动到 iCloud Drive 下（如 `~/Library/Mobile Documents/com~apple~CloudDocs/shanghai-map/`），或至少把 `data/` 放入 iCloud。
2. 其他 Mac 登录同一 Apple ID，等待 iCloud 同步该文件夹。
3. 各设备用 `start.command` 启动，读到的是同步后的同一份收藏。
4. 注意：iCloud 同步是**近实时**（秒级至分钟级），两台设备**同时**写同一份收藏可能产生同步冲突文件；日常错开使用无碍。属个人小范围使用，可接受。

> 手机端：iOS 无法运行 `server.py`，本期不提供手机端交互界面；手机可通过 iCloud 查看/编辑 `profiles.json` 文件本身，或作为未来增强。

## 技术选型

- 地图：Leaflet + GeoJSON（可关闭底图，只画自定义图层）
- 应用形态：本地轻服务（`server.py`，Python 标准库）+ 前端页面；双击 `start.command` 启动
- 抓取：Python（多源可插拔模块架构）
- 成交价主源：上海“网上房地产”/住建局官网网签数据；评价：贝壳点评（可插拔多评价源）；开发商、对口学校为第二批字段
- 多人收藏 + 跨设备：本地多档案身份切换 + iCloud 文件同步（无账号体系）
- 地理数据来源：行政区/街道边界的公开 GeoJSON（如 DataV 等），实现时实际拉取验证

## 范围之外（YAGNI）

**第一阶段不做**（留给第二阶段或以后）：
- 不做云端后端/数据库/线上部署（第一阶段本地轻服务仅限本机）
- 不做手机/平板交互界面与响应式（第一阶段仅桌面 Mac；手机是第二阶段云版的重点）
- 不做实时抓取（手动跑，全量）
- 第一阶段跨设备同步仅用 iCloud 文件同步（限苹果设备），不自建同步服务

**始终不做**：
- 不做带密码的用户账号体系（多人一律无密码身份选择）
- 板块下钻切换视图（方式 2）作为未来可选增强
